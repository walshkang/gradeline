# Gradeline CLI Feedback & Improvements Log

This document records usability pain points and suggested feature improvements discovered during grading setup sessions.

---

## Usability Pain Points & Feature Requests

### 1. Hard Dependency on PDF for Solutions
* **Issue**: The tool strictly expects a PDF file for the solutions key (`solutions.pdf`). Instructors may draft answer keys in MS Word (`.docx`), resulting in manual conversion steps.
* **Impact**: Friction during assignment initialization. If the user forgets to export to PDF, the tool fails during LLM upload.
* **Suggested Solution**: 
  * Add support for native text/Markdown solution keys (`solutions.txt`, `solutions.md`).
  * Integrate a lightweight Python-based `.docx` to text parser (e.g., using `python-docx`) to read MS Office files directly for LLM prompting, bypass-converting to PDF if no visual markup is needed.
* **Status/Roadmap**: Placed in Backlog (`BL-DOCX` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 2. Strict Interactive Terminal (TTY) Check for Wizards
* **Issue**: Workflow commands like `quickstart` and `setup` require a TTY check (`sys.stdin.isatty()`) and fail in piped or agentic environments.
* **Impact**: Agents/scripts cannot automate profile creation easily without manually writing files.
* **Suggested Solution**: 
  * Add a `--non-interactive` or `--auto-accept` flag that bypasses prompt loops and generates default configs directly.
* **Status/Roadmap**: ✅ Completed in Wave 2 (`W2-TTY` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 3. Sub-Optimal Candidate Search in Downloads
* **Issue**: The scan algorithm inside `Downloads` is highly sensitive to older assignment names. For example, during a `hw2` setup, it recommended old `HW1` files as candidates simply because they were present.
* **Impact**: Confusion during wizard execution.
* **Suggested Solution**: 
  * Incorporate date sorting (preferring the most recently modified files) alongside keyword matching.
  * Weight matches higher if they contain the exact string of the current profile name.
* **Status/Roadmap**: Placed in Backlog (`BL-SEARCH` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 4. Direct Support for D2L Brightspace ZIP File Contents
* **Issue**: When importing, the ZIP contains a flat list of folders which the import command handles, but it also copies irrelevant meta files like `index.html` from the D2L export ZIP.
* **Impact**: Messy `submissions` folder that needs manual cleanup.
* **Suggested Solution**: 
  * The discovery or import module should automatically exclude common metadata files like `index.html` or `index.txt` generated by D2L.
* **Status/Roadmap**: ✅ Completed in Wave 2 (`W2-ZIP` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 5. CPU & Speed Bottlenecks on Large Scanned/Handwritten PDFs
* **Issue**: Students frequently submit high-resolution scans of handwritten homework (e.g., 30–36 MB files with page sizes up to `2160 x 3840` points). Running `pdftoppm` at 216 or 300 DPI on these files produces massive 100+ megapixel images, taking 45–60 seconds per page to render and OCR.
* **Impact**: The pipeline feels stalled or locked up on these submissions, and concurrency gets bottlenecked.
* **Suggested Solution**: 
  * Add a check using `pdfinfo` prior to rendering. If page dimensions in points exceed a reasonable limit (e.g., `> 1000 pt` on any edge), dynamically scale down the DPI so the output image does not exceed `2048px` on its longest edge.
  * Move rendering and local OCR into a decoupled, asynchronous preprocessing phase so it doesn't block the grading thread pool.
* **Status/Roadmap**: ✅ Completed in Phase 1 / Pre-Wave (`F-5` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 6. Repetitive Subprocess Stderr Warnings Polluting the Terminal
* **Issue**: Underlying rendering tools (like `MuPDF` or `pdftoppm`) output verbose warnings to stderr (such as `MuPDF error: format error: No common ancestor in structure tree`). These print directly to the user's terminal.
* **Impact**: CLI output gets cluttered and hard to read, and it makes users think the run is failing.
* **Suggested Solution**: 
  * Suppress or redirect the stderr of these CLI calls to a dedicated `diagnostics.log` file, only surfacing them if the command actually returns a non-zero exit code.
* **Status/Roadmap**: ✅ Completed in Phase 1 / Pre-Wave (`F-6` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 7. Missing Interval Checkpointing during Live Runs
* **Issue**: Currently, the `.gradeline_checkpoint.json` is only saved when the grading run is interrupted, rate-limited, or completes successfully. If the terminal is closed, the system crashes, or the process is killed halfway through a long run, all progress is lost.
* **Impact**: High risk of lost work during long-running batch runs.
* **Suggested Solution**: 
  * Modify `save_checkpoint` to be called incrementally (e.g., every time a student's submission successfully finishes grading and is appended to `self.submission_results`).
* **Status/Roadmap**: ✅ Completed in Phase 1 / Pre-Wave (`F-7` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 8. Potential Resolution Mismatch in Tesseract OCR
* **Issue**: In `run_ocr_all_pages`, `pdftoppm` is called without specifying the `-r` DPI argument (which defaults to `150`), but then `--dpi 300` is passed to `tesseract`.
* **Impact**: This DPI mismatch can reduce local OCR accuracy because Tesseract expects a 300 DPI image but receives a 150 DPI image.
* **Suggested Solution**: 
  * Align the DPI arguments in `run_ocr_all_pages` so both `pdftoppm` and `tesseract` use the same matching resolution.
* **Status/Roadmap**: ✅ Completed in Phase 1 / Pre-Wave (`F-8` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 9. PDF Annotation Overlap Mitigation
* **Issue**: PDF annotations (question results, verdicts, and feedback text) often overlap each other when multiple subparts or adjacent questions map to the same coordinates/page, or when reasons are physically long.
* **Impact**: Unreadable, cluttered annotation overlays on student PDFs.
* **Suggested Solution**: 
  * Implement visual collision detection in `grader/annotate.py`.
  * Track a list of bounding boxes (rects) of placed annotations per page.
  * When placing a new annotation, estimate its dimensions (based on font size and text length) and shift it vertically or horizontally until a non-overlapping slot is found.
* **Status/Roadmap**: ✅ Completed in Wave 1 (`W1-OVERLAP` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 10. Inaccurate Coordinate Resolution for Scanned/Handwritten Submissions
* **Issue**: Visual annotations are often placed at incorrect coordinates on handwritten submissions (like Aldo Arossa's). Scanned PDFs frequently lack a clean text layer (rendering text-based local anchors useless) and may contain page rotation (e.g. 90/180/270 degrees) or custom CropBoxes that the coordinate mapping does not adjust for.
* **Impact**: Marks and annotations are printed outside the visible margin or on incorrect areas of the page.
* **Suggested Solution**:
  * Retrieve and respect `page.rotation` and actual page boundaries (CropBox/MediaBox) from PyMuPDF in `resolve_model_location` and coordinate conversion functions.
  * If a page is rotated, apply a geometric transformation to coordinate mapping.
  * Provide fallback locator heuristics or visual margin anchors for scanned documents lacking embedded OCR text.
* **Status/Roadmap**: ✅ Completed in Wave 1 (`W1-COORD` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 11. PDF Viewer Viewport Scrolling and Centering Focus
* **Issue**: The PDF viewer panel in the web UI is large, and when clicking on a question number in the right-hand nav drawer, the page loads but does not scroll to focus on the question. The user has to manually scroll, and the target coordinates/marker can be far down.
* **Impact**: Extra friction when reviewing submissions question-by-question.
* **Suggested Solution**:
  * In the web app (`app.js`), after calling `loadCurrentPage()` and `renderMarker()`, automatically scroll the `.image-wrap` container so the marker is focused near the top of the viewport (at the same height as the toolbar/buttons), rather than staying at the top of the page or in a random scroll position.
* **Status/Roadmap**: ✅ Completed in Wave 1 (`W1-UX` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 12. Review State Visibility and Control in Matrix View
* **Issue**: The Matrix View is useful for spotting anomalies, but it does not display whether a question has been marked as "Reviewed" (synced with the review view's "Mark Question as Reviewed" checkbox). Additionally, there is no way to toggle the review status of a cell directly from the Matrix panel.
* **Impact**: Hard to track review progress across all students and questions at a glance.
* **Suggested Solution**:
  * Update `get_matrix` in `grader/review/api.py` to include the `"reviewed"` field in the response cells payload.
  * In the frontend Matrix Grid, add a visual badge (e.g., green dot, small checkmark icon, or border style) to cell containers that have been reviewed.
  * In `showMatrixDetail`, add a "Reviewed" toggle checkbox/button that directly patches the review state (`reviewed_final`) via API and re-renders the Matrix View.
* **Status/Roadmap**: ✅ Completed in Wave 1 (`W1-MATRIX` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).


### 13. Transparency on Review Export Artifacts
* **Question**: What does the export button in the web UI do, and what files are saved where?
* **Answer / Documentation**:
  * Clicking **Export** triggers the `/api/export` endpoint on the backend, which executes `export_review_outputs` in [exporter.py](file:///Users/walsh.kang/Documents/GitHub/gradeline/grader/review/exporter.py).
  * It generates/saves the following files inside the `<output_dir>/review/` directory:
    1. **Reviewed PDFs**: Student submissions are annotated with final reviewed grades/marks and saved to `review/reviewed_pdfs/` (using the same directory structure and filenames).
    2. **Review Decisions JSON**: A snapshot of the review state and decisions is saved to `review/review_decisions.json`.
    3. **Grading Audit CSV**: Updated grading audit entries representing reviewed results saved to `review/grading_audit_reviewed.csv`.
    4. **Review Queue CSV**: Updated queue records saved to `review/review_queue_reviewed.csv`.
    5. **Brightspace Grades Import CSV**: A ready-to-import CSV with final scores mapped from the grades template spreadsheet, saved to `review/brightspace_grades_import_reviewed.csv`.
* **Status/Roadmap**: ✅ Completed in Wave 1 (`W1-EXPORT` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).

### 14. LLM Run Cost Breakdown
* **Feature Request**: Track and display a detailed cost breakdown of LLM API usage per run, per student, and per question.
* **Suggested Solution**:
  * Parse and capture `usage_metadata` (input tokens, output tokens, cached tokens) from Gemini API response payloads in `gemini_client.py`.
  * Store token usage per question in the results data structure and save to `grading_audit.csv` / `review_state.json`.
  * Implement a billing helper that maps token counts to model pricing (e.g., input and output prices for Gemini 1.5 Flash).
  * Display a cost summary in the CLI output summary bands and in the review server's "Run Summary" dashboard.
* **Status/Roadmap**: Addressed in Future Wave 4 (`W4-COST` in [roadmap.md](file:///Users/walsh.kang/Documents/GitHub/gradeline/docs/plans/roadmap.md)).

### 15. CLI Run Summary Subparts Resolution
* **Issue**: The CLI run summary (e.g., in `regrade` outputs) displays progress or percentages, but doesn't fully represent the status of individual subparts (like `7.A`, `7.B`, `8.A`, `8.B` or the details of sub-questions). We should fix the summary metrics/display to clearly reflect subparts as well.
* **Suggested Solution**: Update the summary output logic to aggregate and display details for subparts alongside parent questions, rather than collapsing them or omitting subpart-level resolution metrics.
* **Status/Roadmap**: *Active Feedback* - To be addressed in CLI/grading updates.

### 16. PDF Annotation Overlap with Student Text
* **Issue**: PDF annotations (verdicts, grades, and feedback comments) sometimes overlap egregiously with the student's own handwritten or typed text/answers on the document, making both the student's work and the feedback hard to read.
* **Suggested Solution**: 
  * Extract text bounding boxes or visual bounding boxes (for scans/handwritten work) around the student's content on the target page.
  * Adjust annotation placement coords dynamically to sit in the margins, whitespace, or unoccupied space near the target coordinates instead of directly overlaying text elements.
* **Status/Roadmap**: *Active Feedback* - To be scoped (potentially as part of layout/spacing improvements).
