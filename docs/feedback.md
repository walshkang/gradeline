# Gradeline CLI Feedback & Improvements Log

This document records usability pain points and suggested feature improvements discovered during grading setup sessions.

---

## Usability Pain Points & Feature Requests

### 1. Hard Dependency on PDF for Solutions
* **Issue**: The tool strictly expects a PDF file for the solutions key (`solutions.pdf`). Instructors often draft answer keys in MS Word (`.docx`), resulting in manual conversion steps.
* **Impact**: Friction during assignment initialization. If the user forgets to export to PDF, the tool fails during LLM upload.
* **Suggested Solution**: 
  * Add support for native text/Markdown solution keys (`solutions.txt`, `solutions.md`).
  * Integrate a lightweight Python-based `.docx` to text parser (e.g., using `python-docx`) to read MS Office files directly for LLM prompting, bypass-converting to PDF if no visual markup is needed.

### 2. Strict Interactive Terminal (TTY) Check for Wizards
* **Issue**: Workflow commands like `quickstart` and `setup` require a TTY check (`sys.stdin.isatty()`) and fail in piped or agentic environments.
* **Impact**: Agents/scripts cannot automate profile creation easily without manually writing files.
* **Suggested Solution**: 
  * Add a `--non-interactive` or `--auto-accept` flag that bypasses prompt loops and generates default configs directly.

### 3. Sub-Optimal Candidate Search in Downloads
* **Issue**: The scan algorithm inside `Downloads` is highly sensitive to older assignment names. For example, during a `hw2` setup, it recommended old `HW1` files as candidates simply because they were present.
* **Impact**: Confusion during wizard execution.
* **Suggested Solution**: 
  * Incorporate date sorting (preferring the most recently modified files) alongside keyword matching.
  * Weight matches higher if they contain the exact string of the current profile name.

### 4. Direct Support for D2L Brightspace ZIP File Contents
* **Issue**: When importing, the ZIP contains a flat list of folders which the import command handles, but it also copies irrelevant meta files like `index.html` from the D2L export ZIP.
* **Impact**: Messy `submissions` folder that needs manual cleanup.
* **Suggested Solution**: 
  * The discovery or import module should automatically exclude common metadata files like `index.html` or `index.txt` generated by D2L.

### 5. CPU & Speed Bottlenecks on Large Scanned/Handwritten PDFs
* **Issue**: Students frequently submit high-resolution scans of handwritten homework (e.g., 30–36 MB files with page sizes up to `2160 x 3840` points). Running `pdftoppm` at 216 or 300 DPI on these files produces massive 100+ megapixel images, taking 45–60 seconds per page to render and OCR.
* **Impact**: The pipeline feels stalled or locked up on these submissions, and concurrency gets bottlenecked.
* **Suggested Solution**: 
  * Add a check using `pdfinfo` prior to rendering. If page dimensions in points exceed a reasonable limit (e.g., `> 1000 pt` on any edge), dynamically scale down the DPI so the output image does not exceed `2048px` on its longest edge.
  * Move rendering and local OCR into a decoupled, asynchronous preprocessing phase so it doesn't block the grading thread pool.

### 6. Repetitive Subprocess Stderr Warnings Polluting the Terminal
* **Issue**: Underlying rendering tools (like `MuPDF` or `pdftoppm`) output verbose warnings to stderr (such as `MuPDF error: format error: No common ancestor in structure tree`). These print directly to the user's terminal.
* **Impact**: CLI output gets cluttered and hard to read, and it makes users think the run is failing.
* **Suggested Solution**: 
  * Suppress or redirect the stderr of these CLI calls to a dedicated `diagnostics.log` file, only surfacing them if the command actually returns a non-zero exit code.

### 7. Missing Interval Checkpointing during Live Runs
* **Issue**: Currently, the `.gradeline_checkpoint.json` is only saved when the grading run is interrupted, rate-limited, or completes successfully. If the terminal is closed, the system crashes, or the process is killed halfway through a long run, all progress is lost.
* **Impact**: High risk of lost work during long-running batch runs.
* **Suggested Solution**: 
  * Modify `save_checkpoint` to be called incrementally (e.g., every time a student's submission successfully finishes grading and is appended to `self.submission_results`).

### 8. Potential Resolution Mismatch in Tesseract OCR
* **Issue**: In `run_ocr_all_pages`, `pdftoppm` is called without specifying the `-r` DPI argument (which defaults to `150`), but then `--dpi 300` is passed to `tesseract`.
* **Impact**: This DPI mismatch can reduce local OCR accuracy because Tesseract expects a 300 DPI image but receives a 150 DPI image.
* **Suggested Solution**: 
  * Align the DPI arguments in `run_ocr_all_pages` so both `pdftoppm` and `tesseract` use the same matching resolution.
